Requesting a new tag for an existing package

To request a tag, please do the following things:

  • Make sure it compiles in all the nightlies you request it in (CMT + RootCore, e.g. AthAnalysisBase and EventLoopAnalysisBase - remember both!)
  • Run your own tests that will run in the nightlies to make sure they pass (CMT instructions, for RootCore: 'rc test_ut --package=YourPackage')
  • Are you changing any interfaces that could affect client packages? If so, the clients need to be warned and given a chance to adjust before the tag goes in.
  • Does your new tag have a potential impact on existing code results? Have a brief statement prepared about how results might change with existing code.

Once you have this information, open a jira ticket in the ATLASG project, ensuring that you select the release requests component for the ticket.

If you hit any problems or have any questions, email the release-help mailing list: atlas-sw-pat-releasehelp@cern.ch

Requesting new packages

You need to make a presentation at the ASG general meeting to say:
  • What the purpose of your package is
  • Who will use it
  • How big it is
  • How do users use it (i.e. explain the methods of the interfaces)

See https://indico.cern.ch/event/532191/contributions/2167754/attachments/1273075/1887639/ArthurLesage_ASGMeeting_20160513.pdf for an example of such a presentation

To determine the size of your package, you can do this with the rootcore build system using: 

rc compile
 rc strip --remove-debug
 rc du

Running ATN tests for the cmt analysis releases

It is essential that any new tag you request does not cause a failure of an ATN test. ATN tests run automatically in the nightly system. See here for the test results of the nightly.

If our release shifter tells you your package request is causing an ATN test failure, your request will be rejected and you will need to fix the test before your request will be accepted again.

Before submitting a package request, you should ensure the tests of that package work. To do this, assuming you have the package already checked out and compiled, you can follow instruction at https://twiki.cern.ch/twiki/bin/view/AtlasProtected/AthAnalysisBase#How_to_execute_atn_tests_locally to execute the tests. You should address any failures before requesting new package tags.

If a release shifter says you have caused an ATN test failure, they should point you are the test, which will have a name, for example "TrigSteeringEvent#TrigSteeringEventTest". This means you should look at the TrigSteeringEvent package and look at the TrigSteeringEventTest, in this case it would be here. So this is a makecheck test, which can be tested by checking out the package and running make check from inside the cmt directory. Again, see the link above for details of how to execute tests.
