Note you can also setup eventloop releases using asetup: asetup AnalysisBase,2.3.30 is equivalent to rcSetup Base,2.3.30.

If you are very used to eventloop, or like the syntax for the commands, when you call setupATLAS: 

setupATLAS

You will find you have a set of useful cmt 'actions' available to you after you have setup an athena release (analysis or otherwise): 

asetup AthAnalysisBase,2.3.29
cmt show actions

The second line lists all the 'commands' you will have. You will then be able to do: cmt find_packages, cmt compile. There is also cmt new_skeleton to quickly create a package with an analysis alg and a joboption in it (in the share directory), ready for xAOD analysis

rcSetup notes

Additional commands:

    • rcSetup -u -> unsetup previous version
    • rcSetup -d -> download packages from SVN instead of creating links
    • rcSetup -d -> download packages from SVN instead of creating links
      • Next steps: source setup.sh && find_packages && source setup_external.sh
    • rc version -> version numbers of all packages in the release
    • rc checkout_pkg -> checkout package from SVN, full path (see rc version) can contain tag

Usage of rcSetup

The script rcSetup is available on both CERN AFS and CVMFS, and is accessible by doing one of the following:

  1. rcSetup after running setupATLAS (recommended, using CVMFS)
  2. alias rcSetup="source /afs/cern.ch/atlas/software/ASG/scripts/rcSetup.sh" (using AFS)
  3. alias rcSetup="source /afs/cern.ch/atlas/software/ASG/scripts/cvmfs/rcSetup.sh" (using CVMFS)
For doing analysis via grid submission it is recommended to set up using cvmfs.

Note rcSetup supports bash/zsh and tcsh. This does not work for sh or you will get complaints about: setup_external.RootCore.sh: file not found

Its usage is as follows: 

lxplus% rcSetup -h

Usage: rcSetup [options] project,[nightlyType,]release,[,configs]

       where project/release/configs are delimited by "," or space:
          Base | Top | SUSY   # project choice, such as "Base"
          nightlyType        # type of nightly, such as "dev"
          64 | 32            # 64-bit (x86_64, the default) or 32-bit (i686)
          slc5 | slc6        # using slc5 or slc6
          opt | dbg          # using opt (default) or dbg version of ASG releas
          gcc43 | gcc46 | gcc47  # in format gcc* set gcc version

The default configuration can be overridden by defining env var "rcSetupSite".


Options:
  -h, --help          show this help message and exit
  -d                  Download source code of a release, try '-d -h' for usage
  -b, --buildRelease  download and build release under current directory
  -V, --version       Print rcSetup version
  -r, --releases      Print available releases
  -c, --printConfigs  Print available config(s) for given project/release
  -f, --findPackages  Automatic running 'rc find_packages' in this setup
  -u, --unSetup       unSet currently existing ASG environment

Setup of previous Analysis release

One useful feature in rcSetup is, that you can set up your previous ASG release env under the same working directory simply by running rcSetup without any argument if you use ROOT from cvmfs, i.e. 

rcSetup

after running "AtlasSetup", or simply 

source rcSetup.sh

even without the need of running "AtlasSetup".

Downloading/Building a release locally

In case you have Internet, but do not have access either CVMFS or CERN AFS, or you like to work on a platform (says Mac OS X) not currently supported, you can still using rcSetup to download the full source code for a given release and build the release yourself.

Downloading rcSetup

You can download rcSetup and define alias for rcSetup: 

svn co svn+ssh://svn.cern.ch/reps/atlasoff/rcSetup/trunk rcSetup
alias rcSetup="source $PWD/rcSetup/rcSetup.sh"

Its usage is as follows: 

rcSetup -d -h

Usage: rcSetup [options] project,release

       where project/release are delimited by "," or space:


Options:
  -h, --help          show this help message and exit
  -d                  Download source code of a release
  -q, --quiet         Do not ask question, use default answer
  -V, --version       Print rcSetup version
  -l, --listPackages  Print list of packages in a given release
  -r, --releases      Print available releases for download
  -b, --buildRelease  download and build release under current directory

Native installers for Linux/MacOS

If you are running Linux or MacOS, you can use native installers to install one or more versions of AnalysisBase:

  • http://cern.ch/akraszna/AtlasAnalysisReleaseInstaller (Linux)
  • http://cern.ch/akraszna/AtlasAnalysisReleaseInstaller.dmg (macOS)

These make it very simple, as they have a relatively up-to-date list of releases from which you can choose. The code responsible for generating the releases is at: https://gitlab.cern.ch/akraszna/AtlasInstaller

Note: In order to bypass macOS's system protections (since the installer application is not a "signed" binary), your best bet is to download the DMG file from the terminal with: 

curl https://akraszna.web.cern.ch/AtlasAnalysisReleaseInstaller.dmg --output AtlasAnalysisReleaseInstaller.dmg

Since the resulting binary will be handled by the system as something that the user created, and not downloaded from the internet. And hence macOS will allow you to execute it with no questions asked. :-P

First installation

  1. Simply navigate to the urls above, then run the downloaded executable and follow the installer instructions. Note: Mac-users see the Mac-specific instructions below if you have trouble with security settings.
  2. Choose an installation folder
  3. Tick the list of release(s) and/or components you'd like to install (externals, release and CMake)
  4. Accept the bundled software licenses.
The installation takes about 1GB of space (for AnalysisBase,21.2.43).

Supposing you chose the location ~/ATLAS/Software as the installation location, then you can set up a release as follows: 

source ~/ATLAS/Software/AnalysisBase/[release_number]/InstallArea/*/setup.sh

It may be convenient to define a function in your .bashrc or .profile file as follows: 

absetup() {
    # Remember the release number
    RELEASE=$1
    # Set up the release
    source ~/ATLAS/Software/AnalysisBase/${RELEASE}/InstallArea/*/setup.sh
    echo "Configured AnalysisBase from: ${AnalysisBase_DIR}"
    # Clean up
    unset RELEASE
}

If installing a different release (e.g. AnalysisTop), then substitute the release name for "AnalysisBase" above.

You can inspect the code in ~/ATLAS/Software/AnalysisBase/[release_number]/InstallArea/*/src.

Updating the installation

Once you have installed the ATLAS releases once, should you wish to modify your installation (add another release, remove an existing release), you will find a "maintenancetool" executable in ~/ATLAS/Software. Run this, which will open a wizard allowing you to make modifications of this nature.

Specific instructions for Mac

Opening the dmg will give you a disk image with the AtlasAnalysisReleaseInstaller executable. MacOS security will prevent you from running this by double-clicking, so right-click and select "Open", then follow the dialogue box.

Downloading source code of a release

To download Base-2.0.12, just run 

  rcSetup -d Base,2.0.12

which would download all the required packages for Base-1.0.0, at the end print out a list of package tags downloaded, and a list of packages not downloaded because they already exist under the working directory.

Then you can compile all the packages via command "rc compile".

Building a release locally

If you like to use your local release in the future, you can build the release via command "rcSetup -b". For example, to build ASG release of Base-1.0.0, just run 

  rcSetup -b Base,2.0.12

which will

  1. create directory $PWD/AnalysisBase/2.0.12/,
  2. download the packages (if not downloaded yet) in ASG release Base-2.0.12,
  3. compile the release,
  4. write a private configuration file $HOME/.rcSetup.conf if the above compilation is successful.

Later you can just run "rcSetup Base,2.0.12" to use this local ASG release.

Note: this feature is still under development stage.

How to submit to the grid

You can submit you ROOT job to the grid, using your usual prun command, and adding --useRootCore. For example: 

prun --useRootCore
        --inDS=mc12_8TeV.157814.Sherpa_CT10_VVtoeeqq.merge.NTUP_SUSY.e1515_s1499_s1504_r3658_r3549_p1328/ --outDS=user.username.ASGtest_v11
        --nGBPerJob=14
        --writeInputToTxt IN:input.txt
        --exec "macrotest" 
        --site=ANALY_AUSTRALIA

The job will only compile the users package, and the rest is taken from the release. Note your analysis release must be using RootCore-00-02-56 or newer.
